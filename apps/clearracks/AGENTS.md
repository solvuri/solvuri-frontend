## ClearRacks — project notes

- This app owns its own cart state (`lib/store.ts`, Zustand + `localStorage` persistence under `clearracks-storage`). Don't move it back into a shared package unless a second app genuinely needs the same cart shape.
- Storefront routes live under `app/storefront/[subdomain]/...` and are reached via `proxy.ts`'s subdomain rewrite (renamed from `middleware.ts` — Next.js 16's non-deprecated convention). It treats any `localhost:<port>` as the root domain, so subdomain routing works whether this app runs on its own dev port (3001) or alongside the others.
- The storefront listing page (`useProducts()`), product detail page (`useProduct(id)`), and order detail page (`orders/[id]/page.tsx`, via `useOrder(id)`) all fetch through `@repo/data`, currently backed by mock data (`packages/data/src/mock/products.ts`, `packages/data/src/mock/orders.ts`) — swap the corresponding `fetch*` function in `packages/data/src/products.ts`/`orders.ts` for a real API call when one exists; no consumer changes needed. Note: the orders *list* page (`orders/page.tsx`) still has its own separate, richer mock dataset (including "SAF-"/"SVC-" experience/service bookings that don't fit the `Order` shape) — only the two "ORD-" product-order IDs there resolve via `useOrder`; the others 404 if clicked through to details.
- Client Component pages under `app/` receive `params` as a `Promise` in this Next.js version, not a plain object — unwrap it with React's `use(params)`, not `params.foo` directly (the sync-access shim from Next 15 is gone). Got this wrong once already wiring up the product detail page above; double-check any new dynamic route page. `product/[productId]/page.test.tsx` and `orders/[id]/page.test.tsx` are regression tests for exactly this — if a dynamic route page reverts to sync `params` access, those tests fail. When testing a page that calls `use(params)`, wrap the initial `render()` in an awaited `act()` (see either test file) — otherwise the component's initial suspend happens outside what `render()` awaits and the test hangs on the Suspense fallback.
- `pnpm test` here runs `lib/store.test.ts` (cart logic) plus the two dynamic-route regression tests above via Vitest + Testing Library + jsdom (`vitest.config.ts`).
- `Product` types come from `@repo/types` — extend that shared interface rather than hand-rolling a new product shape per component.
- Shared design tokens (background/surface/foreground/border/ring/radius/fonts) come from `packages/ui/src/styles/solvuri-theme.css`'s vocabulary; this app supplies its own ClearRacks brand values in `app/globals.css`, distinct from the SOLVURI palette used by `web`/`admin-portal`.
- `components/navigation/Navbar.tsx`/`Footer.tsx` (the marketing chrome, used by the `(marketing)` layout) are thin wrappers around `@repo/ui`'s shared `Navbar`/`Footer` (`variant="sticky"`) — the actual markup lives in `packages/ui/src/navbar.tsx`/`footer.tsx`'s `StickyNavbar`/`StickyFooter` branches. The storefront's `components/stores/navigation/StoreNavbar.tsx`/`StoreFooter.tsx` are deliberately **not** part of this — they're cart-aware, per-tenant storefront chrome, a different component solving a different problem, and stay local to this app.
